03 / 07 · Stack

El cómo

Un proceso · un lenguaje en el servidor · cero build
i.
Backend
FastAPI
Pydantic valida en el borde. OpenAPI se genera del código. El contrato no se desincroniza.
ii.
Persistencia
SQLite + SQLAlchemy
Un archivo, sin servidor de BD. ORM declarativo, cascadas explícitas en el modelo.
iii.
Frontend
HTML · CSS · JS
Vanilla. Sin framework, sin bundler, sin transpilador. Lo que escribo es lo que el navegador ejecuta.
iv.
Servidor estático
El mismo FastAPI
StaticFiles sirve el frontend. Mismo origen, sin CORS, un solo proceso para levantar.

05 / 07 · Justificación

El por qué

Tres principios detrás de cada elección
Principio 01

Contrato vivo

FastAPI deriva la documentación OpenAPI del código. El contrato cliente-servidor no es un documento aparte que se desactualiza, es el código mismo.

Principio 02

Cero ceremonia

Sin Docker, sin bundlers, sin migraciones, sin orquestación. Menos piezas significa menos superficie donde la demo puede fallar y menos lugares donde el agente puede equivocarse.

Principio 03

Tipos explícitos

Pydantic y los type hints de Python reducen el margen que tiene la IA para inventar. Cuando los tipos están escritos, el código generado es más predecible.

Domar al agente

Lo que la consigna realmente evalúa

El rol es de arquitecto, no de tipeador.

Cada archivo de configuración existe para anclar al modelo a un plan y darle al evaluador algo concreto sobre lo que preguntar en la defensa.

  • CLAUDE.md Arquitectura, contrato API, convenciones de naming, reglas explícitas para el agente.
  • settings.json Restricciones de permisos. Operaciones destructivas requieren confirmación.
  • /skill Skill propia que automatiza el patrón de agregar una entidad nueva al sistema.
